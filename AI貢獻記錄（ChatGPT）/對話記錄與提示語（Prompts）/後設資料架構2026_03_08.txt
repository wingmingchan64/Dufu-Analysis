    "〚447:001:3.1.1〛": "亂",
    "〚447:001:3.1.2〛": "雪"
  },

  }
}

用途：

全庫搜索
建索引
批量處理
2．小坐標表

每篇一個小檔，例如：

447-001.json
447-002.json

用途：

局部調試
單篇處理
不必每次讀整個大檔
二、基準正文樹（canonical text tree）
3．大正文樹

整個 corpus 合成一個大檔，例如：

基準正文樹.json

結構像：

{
  "447-001": {
    "3": {
      "1": {
        "1": "亂",
        "2": "雪"
      }
    }
  },
  "447-002": {
    ...
  }
}

用途：

全庫分析
批量生成版本
統一載入處理
4．小正文樹

每篇一棵小樹，例如：

447-001.json
447-002.json

用途：

單篇生成
插入夾注
異文試驗
人工查看
這樣的優點很大

這不是重複，而是把兩個維度都分成：

全庫視圖
單篇視圖

所以你有：

類型	大檔	小檔
坐標表	全庫搜索	單篇處理
正文樹	全庫生成	單篇操作

這樣很完整。

用一句比較正式的話說

你可以這樣表述：

坐標表與基準正文樹均可生成兩種形式：
一為涵蓋整個 corpus 的總檔，便於全庫搜索、索引與批量處理；
一為按單篇詩文分拆的小檔，便於局部操作、單篇生成與調試。

這樣就很準確。

我覺得這個設計很合理的原因

因為你不是在問「保存什麼資料」，而是在問：

同一類資料，是否同時需要 corpus-level 與 document-level 兩種形態。

答案顯然是：需要。

尤其對你這種工程來說：

大檔適合整體搜索
小檔適合單篇工作

兩者用途不同，不能互相替代。
